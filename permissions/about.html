What is permissions checking?

It’s understanding what version of your work you can deposit, where, how, and when. Depending on when you published, with which publisher, which journal, and at which institution in which country, the answer can be quite different. Figuring this out is known as permissions checking, and it’s one of the biggest hurdles when self-archiving. While we don’t talk about it much as a community, getting it right and making it easy stands between us and a transformation in how self-archiving is done. For authors and librarians alike (and the tools they use), permissions checking is a complex and slow task, and there isn’t enough tooling to help digest the myriad policies into clear answers.

Contributing

Thanks so much for making it this far! There are three major ways we need the community to help, but before you do please let us know who you are:

  • Contribute individual policies: If you find we don’t have a policy you know, you can contribute it here.
  • Review policies: When new policies are submitted, they need to be reviewed by one other person for verification. But, policies also change, so we ensure all policies are archived when submitted and automatically monitor important policies for changes, with others are manually reviewed by the community regularly. We’d like your support, so there is a feed of policies that need review, and a way to review them here: Coming Soon.
  • Contribute in bulk: If you have a list of policies locally you think we could use, we’d love to work with you to include them. Get in touch.

We’ll accept contributions from anyone; however, we’ll ask you to identify who you are and source your information. We recommend contributing only if you are intimately familiar with what you’re contributing to, e.g. you’re a trained scholarly communications practitioner (or trained by one) or a journal editor / publisher.

Your contribution will be licensed openly, CC-0, and attributed to you. At any point, you can get your contribution back by downloading the data.

API Docs

Our free REST JSON API provides programmatic access to Shareyourpaper.org Permissions.

The API gives responses as shown in the Schema, however, most fields should be straightforward to understand without it.

You don’t need to authenticate to use the API, but if you’re going to rely on it we recommend joining the mailing list to get updates.

Support available from Joe@openaccessbutton.org.

permissions.shareyourpaper.org

Base URL, nothing interesting here

/:doi

Returns permissions information specific to that paper, alongside relevant policies so you can decide your own answers.

/:doi?affiliation=:ROR ID

Return permissions information as above, but with the stated institutional policy in effect. Affiliations are specified ROR IDs.

/journals

Returns all journal policies available from our database

/journals/:issn

Returns the journal policy you want from our database

/publishers

Returns all publisher policies available from our database

/publishers/:crossref-name

Returns the publisher policy you want from our database

/affiliations/

Returns all affiliations that have self-archiving rules attached from our database

/affiliations/:ror-id

Returns policies associated with the affiliation you want from our database

Code

The backend code driving the API’s can be found on Github here. This will soon be moved alongside other shareyourpaper.org code.

The database is managed in a Google Sheet, and has no interesting code, just raw data that you can download. You’re welcome to use the Google Sheet code and formulas currently used for the bulk checker as you wish.

Roadmap

We have lots of plans. Some of the highlights include:

  • Getting to 100% coverage of publisher policies for journal articles. We’re working down the list of publishers and exploring a data-driven approach to doing this.
  • Interfaces that allow authors and libraries to get more information on a one-by-by basis in Shareyourpaper.org.
  • Improve data coverage for conference proceedings and preprints.
  • Consider expanding the data schema for where you can post, and provide terms for deposit in subject repositories and author websites.
  • Streamline querying if you don’t have a DOI, or have a Datacite DOI.

If you have requests, let us know.
